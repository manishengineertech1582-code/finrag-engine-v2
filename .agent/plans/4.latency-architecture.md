# Plan 4 — Latency & Architecture Hardening

## Objective

Eliminate redundant computation, race conditions, and missing observability that
cause 40–50 s per-query latency and poor UX under rapid successive queries.

---

## Root Cause Summary

| ID | Area | Problem | Fix |
|----|------|---------|-----|
| L1 | `src/retriever.py` | `OpenAIEmbeddings` instantiated on every reranker call — cold-start HTTP client per query | `@lru_cache`-backed `_get_embeddings(model)` factory |
| L2 | `src/retriever.py` | `_SafeEmbeddingsFilter` called deprecated `.get_relevant_documents()` on base retriever | Replaced with `.invoke()` |
| L3 | `src/retriever.py` | `MultiQueryRetriever` generated 3 query variants (LangChain default) — 3× LLM cost | Reduced to 2 via `num_queries` param; exposed as `settings.multi_query_num_queries` |
| L4 | `app/routes/query.py` | No query result cache — identical questions re-ran full pipeline every time | TTL-aware `query_result_cache` in `app/state.py`; hit/miss/expiry logic in query route |
| L5 | `app/routes/query.py` | No `request_id` on responses — impossible to correlate slow responses in logs | `uuid.uuid4().hex` per call; included in `QueryResponse` and log line |
| L6 | `app/routes/query.py` | No latency tracking — no visibility into per-query pipeline cost | `time.perf_counter()` around `qa_chain.invoke()`; returned as `retrieval_meta.latency_ms` |
| L7 | `frontend/src/lib/api.ts` | `api.query()` had no `AbortSignal` support | Added optional `signal?: AbortSignal` parameter threaded to axios |
| L8 | `frontend/src/lib/hooks/useQuery.ts` | No request cancellation — stale responses from old queries could overwrite newer results | `AbortController` ref; cancel-before-new-issue; stale response discard guard |
| L9 | `frontend/src/features/rag/QueryPage.tsx` | Single static loading message — no user feedback on which pipeline stage is running | `loadingStage` state (0→1→2) advancing via `setTimeout` at 800 ms / 2400 ms |

## Files Changed

### Backend

- `config/settings.py` — `query_cache_ttl_seconds: int = 300`, `multi_query_num_queries: int = 2`
- `app/state.py` — `query_result_cache: dict[str, Any] = {}`
- `models/response.py` — `RetrievalMeta.latency_ms: Optional[float]`; `QueryResponse.request_id: Optional[str]`
- `src/retriever.py` — `_get_embeddings` lru_cache factory; `.invoke()` in `_SafeEmbeddingsFilter`; `num_queries` param on `_apply_multi_query` and `get_retriever`
- `src/pipeline.py` — pass `multi_query_num_queries=settings.multi_query_num_queries` to `get_retriever`
- `app/routes/query.py` — TTL cache lookup/store; `request_id`; `latency_ms`; `queries_generated` uses settings value
- `app/routes/ingest.py` — `query_result_cache.clear()` alongside `pipeline_cache.clear()`

### Frontend

- `frontend/src/lib/api.ts` — `latency_ms: number | null` on `RetrievalMeta`; `request_id: string | null` on `QueryResponse`; `signal?: AbortSignal` on `api.query()`
- `frontend/src/state/appStore.ts` — `request_id?: string | null`; `latency_ms?: number | null` on `RecentQuery`
- `frontend/src/lib/hooks/useQuery.ts` — full rewrite: `AbortController` ref; stale guard; `request_id` + `latency_ms` passed to store
- `frontend/src/features/rag/QueryPage.tsx` — `loadingStage` state + `useEffect` timers; staged loading messages

### Tests

- `tests/test_latency_hardening.py` — 18 new tests: embeddings cache, deprecated call removed, num_queries default, query result cache hit/miss/expiry/ingest-invalidation/TTL-zero, request_id uniqueness + freshness on cache hit, latency_ms presence, settings defaults + env overrides
- `tests/test_retrieval_pipeline.py` — updated 5 `_SafeEmbeddingsFilter` tests: mock `base.invoke` (not `get_relevant_documents`); patch `src.retriever._get_embeddings` directly
- `tests/test_document_listing.py` — updated 4 `_SafeEmbeddingsFilter` tests: same pattern

## Latency Impact (Estimated)

| Source | Before | After |
|--------|--------|-------|
| `OpenAIEmbeddings` cold start per query | ~200–500 ms | 0 ms (cached) |
| MultiQueryRetriever: 3 → 2 LLM calls | ~2–4 s saved | ~1–2 s saved |
| Query result cache hit | full pipeline (~8–15 s) | < 1 ms |
| Frontend stale overwrite | possible | eliminated |

## Cache Invalidation

Both `pipeline_cache` and `query_result_cache` are cleared in `_run_ingestion()` after
every successful document ingest. This ensures new documents are immediately reflected
in subsequent queries. The TTL (default 300 s) applies only during normal operation
between ingests.

## Risk Notes

- `_get_embeddings` lru_cache survives API key rotation at runtime; requires process restart (acceptable for current architecture)
- Cache-hit responses intentionally return `latency_ms: null` (no pipeline ran)
- `AbortController` abort is silent (`CanceledError` not surfaced to user) — the next submitted query succeeds normally
