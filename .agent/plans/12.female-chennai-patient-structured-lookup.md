# 12. female-chennai-patient-structured-lookup.md

## Goal
Fix missing answers for structured patient queries such as "Provide female patient details who stay in Chennai" in `C:\Users\manis\Downloads\Production-Project\finrag-engine-old` by replacing the fragile semantic-only path with a deterministic tabular lookup flow.

## Root Cause Hypothesis
1. Patient requests are still answered through the generic RAG path.
2. The patient source is tabular CSV data, but retrieval operates on chunked text slices rather than exact rows.
3. CSV content is chunked for embeddings, so row-level filters like gender + city + outcome are inferred by the model instead of executed explicitly.
4. This causes intermittent false "not available" responses even when matching rows exist in the uploaded raw files.

## Fix Strategy
1. Add a structured patient lookup service that scans raw CSV/XLSX uploads for exact row matches.
2. Parse common patient filters from natural-language prompts: gender, city/location phrasing, outcome, and doctor name.
3. Use the structured path for single patient queries and feed more focused row-level context into compound clauses.
4. Add structured logging for scanned files, parsed filters, and matched rows.
5. Add regression tests for Chennai and adjacent patient lookup cases.

## Risks
- Over-eager patient detection could hijack non-tabular questions.
- Raw upload scanning must respect user scoping and source filters.
- Output format must stay compatible with the existing frontend citation UX.
