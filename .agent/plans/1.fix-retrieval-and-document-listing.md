# Fix Plan: Retrieval Failure + Document Listing Persistence

## Root Cause Analysis

### RCA-1 — Retrieval returns 0 sources (CRITICAL)
- **Symptom**: sources=0, confidence=0.10, "not available in provided documents"
- **Cause**: `rerank_threshold=0.75` in `EmbeddingsFilter` drops ALL retrieved chunks
- **Mechanism**: `text-embedding-3-small` query-to-chunk cosine similarity is 0.30–0.65
  for semantically relevant content. 0.75 requires near-verbatim match — virtually nothing passes.
- **Evidence**: "BM25 sees many docs, reranking is applied" → reranker receives docs, drops all
- **Fix**: Lower default to 0.30; add _SafeEmbeddingsFilter with min_docs=1 safety net + per-stage logging

### RCA-2 — Documents vanish after page refresh
- **Symptom**: "No documents uploaded yet" after refresh even though docs exist in vector store
- **Cause**: `appStore.ts` partialize excludes `uploads` from localStorage persistence
  `partialize: (s) => ({ recentQueries: s.recentQueries })` — uploads is session-only
- **Cause**: No `GET /api/documents` endpoint — no backend API to recover the list
- **Fix**: Add `GET /api/documents` endpoint backed by `ingested_files.json`;
  update FileList to merge API data with session uploads

### RCA-3 — Duplicate detection inconsistency
- **Symptom**: Dedup says "already exists" but UI shows nothing (after refresh)
- **Cause**: Dedup registry (`ingested_files.json`) persists; frontend uploads state doesn't
- **Fix**: `GET /api/documents` exposes the registry; dedup response consistent with listing
