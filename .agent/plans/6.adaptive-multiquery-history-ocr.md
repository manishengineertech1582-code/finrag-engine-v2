# Plan 6 — Adaptive MultiQuery + History Cap + OCR UX

## Objective

Three independent production hardening tracks executed in this phase:

1. **Latency** — Adaptive MultiQuery skip for short queries (saves ~1-3s LLM overhead)
2. **History** — Cap chat history to last 5 items; add persistent total counter
3. **OCR UX** — Surface missing Poppler/Tesseract in IngestionPage; wire health OCR types

---

## Root Cause Map

| ID | Area | Problem | Fix |
|----|------|---------|-----|
| RCA-6 | `app/routes/query.py` | MultiQuery runs for ALL questions including single-word lookups; ~1-3s LLM overhead wasted on simple queries | `_is_complex_query()`: skip MultiQuery if < 5 words or < 30 chars |
| RCA-11 | `frontend/src/state/appStore.ts` | `recentQueries` sliced to 20 — history grows unbounded; Dashboard "Questions asked" becomes meaningless | Cap to 5; add `totalQueriesCount` counter that never truncates |
| RCA-9 (UX) | `frontend/src/lib/api.ts` | `HealthResponse` type missing `ocr` field — frontend silently drops OCR availability from health response | Add `OcrDependencies` interface; extend `HealthResponse` |
| RCA-9 (UX) | `frontend/src/features/ingestion/IngestionPage.tsx` | Users uploading scanned PDFs have no indication OCR is unavailable until queries return incomplete results | Amber warning banner on IngestionPage when `!health.ocr.poppler || !health.ocr.tesseract` |

## Files Changed

### Backend

- `app/routes/query.py`:
  - Added `_MULTI_QUERY_MIN_WORDS = 5`, `_MULTI_QUERY_MIN_CHARS = 30` constants
  - Added `_is_complex_query(question: str) -> bool` — public (testable) heuristic
  - `_build_pipeline(request, use_multi_query=True)` — new parameter; cache key now includes `|mq={bool}`
  - `ask_question`: computes `use_multi_query = settings.use_multi_query and _is_complex_query(question)`; passes to `_build_pipeline`; fixes `queries_generated` to use runtime `use_multi_query` not `settings.use_multi_query`
  - Log line updated: now emits `mq=True/False` per query

### Frontend

- `frontend/src/state/appStore.ts`:
  - `HISTORY_LIMIT = 5` exported constant
  - `totalQueriesCount: number` field added (persisted, never truncated)
  - `addRecentQuery` slices to `HISTORY_LIMIT` (was 20); increments `totalQueriesCount`
  - `partialize` persists `totalQueriesCount`

- `frontend/src/lib/api.ts`:
  - `OcrDependencies { poppler: boolean; tesseract: boolean }` interface added
  - `HealthResponse.ocr: OcrDependencies` field added

- `frontend/src/features/ingestion/IngestionPage.tsx`:
  - Uses `useHealth()` to read OCR state
  - Amber warning banner when `!health.ocr?.poppler || !health.ocr?.tesseract`
  - Shows which specific dependencies are missing

- `frontend/src/pages/Dashboard.tsx`:
  - "Questions asked" stat now reads `totalQueriesCount` (persisted, never resets)
  - Sub-label shows "N in history" when queries exist (indicates cap)

### Tests

- `tests/test_adaptive_multiquery_history.py` — 16 new tests:
  - `TestIsComplexQuery` (9 tests): word/char boundary logic, both sides of threshold
  - `TestBuildPipelineAdaptiveMultiQuery` (3 tests): False/True passthrough, cache differentiation
  - `TestQueriesGeneratedInResponse` (3 tests): simple=1, complex=num_queries, global disabled=1

- `tests/test_retrieval_pipeline.py` — already updated in Phase 13

## Latency Impact (Estimated)

| Query type | Before | After |
|------------|--------|-------|
| Short (≤4 words) | MultiQuery ran → ~1-3s LLM overhead | Skipped → 0ms overhead |
| Long (≥5 words, ≥30 chars) | MultiQuery ran correctly (Phase 13 fix) | MultiQuery runs (correct) |

2 LLM expansion calls entirely.

## Risk Notes

- `_is_complex_query` is a conservative heuristic (5 words + 30 chars). Edge cases where
  a short question benefits from expansion (e.g. "EBITDA?") will miss MultiQuery.
  The heuristic can be tuned via `_MULTI_QUERY_MIN_WORDS` and `_MULTI_QUERY_MIN_CHARS`.
- `totalQueriesCount` starts at 0 for existing users (not backfilled from `recentQueries`).
  This is acceptable — it's a forward-counting metric from upgrade time.
- The `bg-amber-500/8` class used in IngestionPage is a Tailwind v4 arbitrary opacity value
  (equivalent to `bg-amber-500` at 8% opacity). Verified in build.
